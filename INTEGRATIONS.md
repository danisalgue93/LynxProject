## Magic Link

Implemented:

- Magic SDK is loaded in `frontend/index.html`.
- Email magic link login uses `VITE_MAGIC_PUBLISHABLE_KEY`.
- The app no longer simulates a successful login when Magic is missing.
- Successful Magic email login stores a managed local session so the header reflects the logged-in state.
- Managed Magic sessions use a stable `MAGIC:<issuer/email>` wallet id for portfolio, notifications, trades, duels, and orderbook calls instead of using the shared demo wallet.
- Logout clears the managed auth session.

Required config in `frontend/.env`:

```env
VITE_MAGIC_PUBLISHABLE_KEY=pk_live_or_pk_test_from_magic
```

Google OAuth note:

- Magic OAuth requires the OAuth extension. The current project uses the CDN SDK path, so email magic link is supported directly.
- For Google OAuth, switch the frontend to package-based Magic SDK usage and install `magic-sdk` plus `@magic-ext/oauth2`.

## MoonPay

Implemented:

- The MoonPay widget URL is now generated by `frontend/server.ts`.
- `npm run dev` now starts `frontend/server.ts`, so local development includes the MoonPay signing route.
- If `MOONPAY_SECRET_KEY` is configured, the widget URL is signed server-side with HMAC-SHA256.
- The client no longer uses a hardcoded `pk_test_key` fallback.
- The wallet screen shows a clear error if MoonPay is not configured.

Required config in `frontend/.env`:

```env
VITE_MOONPAY_API_KEY=pk_test_or_pk_live_from_moonpay
MOONPAY_SECRET_KEY=sk_test_or_sk_live_from_moonpay
```

Do not expose `MOONPAY_SECRET_KEY` in client code or as a `VITE_` variable.
